# ADR 0011 · 部署形态:家用 MacBook + Tailscale 内网,launchd 常驻

- 状态:已采纳(2026-07-17,用户拍板「MacBook 放家里不关机」)
- 关联:P4「部署上线」;`scripts/macos-setup-server.sh`、`scripts/backup.mjs`、[docs/部署.md](../部署.md)

## 背景

应用用 SQLite 单文件库,需要持久磁盘,serverless(Vercel 等)天然不适配。用户为个人使用,选择把家里的 MacBook 当常驻主机。需要决定:进程守护方式、外部访问方式、是否公网、备份方案。

## 决策

1. **进程守护用 launchd,不用 Docker/pm2**。macOS 原生、零额外依赖,开机自启 + KeepAlive 崩溃拉起;脚本生成 plist,幂等可重跑。
2. **外部访问用 Tailscale 内网,不暴露公网**。手机与 MacBook 组网即可随时随地访问;因此 P4「口令登录」降级为公网化的前置项,当前不做。
3. **备份走 better-sqlite3 在线备份 API**(WAL 下运行中备份也一致),连同 `data/uploads` 原图,每日 03:30 落 `backups/`,保留 30 份;异地容灾交给 Time Machine / rclone,不自建。
4. 监听 `0.0.0.0:3000`,同时服务局域网与 Tailscale 网卡。

## 备选与否决理由

- **Docker**:macOS 上要装 Docker Desktop(重、商用许可问题),对单机单应用是纯开销。
- **pm2**:要全局装包且仍需配开机自启,不如 launchd 原生一步到位。
- **VPS**:多一份月费与运维;用户已有常开设备,零成本方案优先,将来要公网再迁移(应用本身无状态耦合,拷 `data/` 即迁移)。
- **cp 冷备份**:WAL 模式下运行中直接 cp 可能拿到不一致快照,故用在线备份 API。

## 影响

- 新增 `scripts/`(backup.mjs、macos-setup-server.sh)与 `npm run backup`;`backups/`、`data/logs/` 加入 gitignore。
- 合盖/休眠会断服务——文档中列为必做配置项(`pmset -c sleep 0`)。
- PWA 加主屏、通知推送等仍在 P4 后续迭代。
