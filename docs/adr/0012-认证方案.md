# ADR 0012 · 认证方案:自建轻量会话 + proxy 中央守卫,共享账本

- 状态:已采纳(2026-07-17,用户要求「先添加登录和注册功能」,提前于公网化)
- 关联:P4「访问保护」;表 `users` `sessions`(迁移 0003);`src/lib/auth*.ts`、`src/proxy.ts`、TC-AU

## 背景

原计划公网化时才做口令登录(ADR 0011),用户要求现在就加登录+注册。需要决定:凭证存储、会话机制、路由保护方式、是否多用户数据隔离。

## 决策

1. **自建,不引 NextAuth 等框架**。个人自托管只需用户名+密码,框架的 OAuth/adapter 体系是纯开销。新增依赖仅 `server-only`(0 运行时代码)。
2. **密码用 Node 自带 scrypt**(随机盐,`salt:hash` 存储,timingSafeEqual 比较),零原生依赖。
3. **会话 = 随机 token 放 HttpOnly cookie(30 天),库中只存 sha256(token)**——库泄露不等于会话泄露,且可逐条吊销;登出即删库中记录。不用 JWT(无法吊销、无收益)。
4. **路由保护走 Next 16 `proxy.ts`(默认 Node runtime,可直查 SQLite)集中拦截**:除 `/login` `/register` `/api/auth/*` 外,页面未登录 302 → /login,API 返回 401。**新增路由默认受保护**,不必逐个记得加守卫。
5. **注册开放但可关**:`ALLOW_REGISTER=0` 关闭注册(自己/家人注册完后建议关闭)。登录失败统一文案「用户名或密码错误」,不暴露用户名是否存在。
6. **共享账本,不做按用户的数据隔离**:所有登录用户看同一本账(家庭场景);登录解决的是「陌生人进不来」。按用户隔离要给全部业务表加 userId、改所有查询,收益与个人定位不匹配——将来真需要再演进。

## 备选与否决理由

- **NextAuth / Auth.js**:为 OAuth 多 provider 设计,配置面大;本场景一张 users 表 + 一张 sessions 表即可。
- **JWT 无状态会话**:登出/吊销做不干净;单机 SQLite 查会话无性能顾虑。
- **逐页面/逐 API 加守卫函数**:27 处调用点,漏一处即漏洞;中央 proxy 是「默认安全」。
- **bcrypt/bcryptjs**:多一个依赖;Node 内置 scrypt 是 OWASP 认可的密码 KDF。

## 影响

- 表 11 张(+users +sessions);E2E 测试默认注入固定会话(setup-db 写入 e2e 用户 + 长效 token,playwright storageState 带 cookie),既有用例零改动;TC-AU 6 例覆盖认证链路。
- 「我的」页显示当前用户名 + 退出登录。
- 部署:升级到此版本后首次访问会要求注册账号;公网暴露的前置条件已具备,但仍推荐 Tailscale 内网(纵深防御)。
