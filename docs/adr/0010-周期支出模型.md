# ADR 0010 · 周期支出模型:下次支付日 + 手动记一笔顺延

- 状态:已采纳(2026-07-16)
- 关联:P3 迭代 2「周期支出识别 + 到期提醒」;表 `recurring`;`src/lib/recurring.ts`

## 背景

订阅、房租等固定支出希望:到期有提醒、入账少按键、能从已有流水里自动发现。需要决定周期建模方式、入账是自动还是手动、候选识别的判定标准。

## 决策

1. **只存「下次支付日」(`nextDate`),不存起始日+规则表达式**。周期仅 `monthly | yearly` 两档。到期判断、提醒都基于 nextDate 与今天的比较(7 天内=即将到期,过期=已到期)。
2. **不自动入账**。用户在管理页点「记一笔」→ 以**今天**为记账日生成一条真实 `transactions` 流水(备注=名称,来源 manual),同时 `nextDate` 顺延一个周期。周期支出不是影子账本,账本里只有真实流水。
3. **顺延月末安全**:目标月没有原始日期时收敛到目标月最后一天(如 1-31 → 2-28)。个人场景接受「31 日订阅在 2 月记于 28 日」的偏移,不额外记忆原始日意图。
4. **候选识别是纯函数启发式**:近 3 个自然月(含当月)每月都出现**同名(备注 优先于 商户)同金额**的支出,且未登记过同名周期支出,即列为候选;建议下次日 = 最后一笔日期 + 1 月。不引入模糊匹配/金额容差。

## 备选与否决理由

- **cron 式规则 / RRULE**:表达力过剩,个人场景 99% 是每月/每年;解析与 UI 成本高。
- **到期自动写入流水**:违背「账本只记真实发生」直觉,自动入账错了要撤销反而更麻烦;且本应用无后台常驻任务,自动化不可靠。
- **金额容差的候选识别(如 ±5%)**:误报会稀释信任;固定支出金额天然稳定,严格相等已够用,不够再放宽。

## 影响

- 新表 `recurring`(无唯一约束,允许同名不同金额并存);迁移 `0002`。
- 首页提醒卡与所选月份无关,始终按「今天」计算;放在预算超支卡之前(可行动性更强)。
- 测试组 TC-RC 5 例;`tests/helpers.ts` resetData 增加 `recurring` 清理。
