# 技术选型决策

> 决策日期:2026-07-12 · 状态:已确认

## 结论

采用 **Next.js 全栈 + SQLite + LLM 视觉识别**(备选方案中的方案 A)。

| 层 | 选型 | 说明 |
|----|------|------|
| 框架 | Next.js(App Router)+ TypeScript | 页面与 API 同一项目,开发效率最高 |
| UI | Tailwind CSS | 手机优先响应式;组件自建为主,保持轻量 |
| 数据库 | SQLite + Drizzle ORM | 单文件数据库,备份=拷贝文件;个人量级绰绰有余 |
| OCR / 识别 | 视觉大模型 API(默认 Claude API,可换) | 截图直接转结构化 JSON(金额/商户/时间/分类、基金字段),同时承担分类推测;不采用传统 OCR |
| 图片存储 | 本地文件系统(`data/uploads/`) | OCR 原图按需求保留,支持设置保留期限 |
| 部署 | 待定 | 先按本地开发环境搭;Docker 化留到部署方式确定时 |

## 被否掉的备选与原因

- **方案 B(Vue + FastAPI + 本地 PaddleOCR)**:OCR 免费且图片不出本机,但"识别文字 → 结构化账目"的解析规则需自行编写维护,支付宝/微信界面改版即失效,开发与维护成本最高。
- **方案 C(纯前端 PWA)**:零部署,但数据锁在单一浏览器,不支持多设备同步,与长期记账场景冲突。

## 关键取舍说明

1. **LLM 视觉 API 替代传统 OCR**:中文支付截图/基金持仓截图版式多变,传统 OCR 需要逐版式写解析;视觉模型直接输出带字段的 JSON,准确率与开发效率都更优。代价是按量付费(个人记账量级约几元/月)与截图需经 API——需求文档 5 中「所有识别结果须人工确认后入账」作为质量兜底。
2. **SQLite 而非云数据库**:单用户自托管,数据归属个人(需求 1.2);未来若上多设备部署,SQLite 文件随服务端走,不需要改动应用层。
3. **API Key 管理**:密钥仅存服务端环境变量(`.env.local`,不入库);前端不接触密钥。

## 环境要求

- Node.js ≥ 20,npm 或 pnpm
- 视觉模型 API Key(运行时通过环境变量注入)
